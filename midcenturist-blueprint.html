01

System Architecture

Three deployable units — the storefront, the CMS admin, and the Flask API — all pointing at the same PostgreSQL database. External services (payment gateways, Instagram API, email) communicate with Flask only. The frontend never touches the database or payment providers directly.

Core principle

The Flask API is the single source of truth. Both the storefront and the CMS are API consumers — they never write to the database directly. All business logic, validation, security, and payment handling lives in Flask. This means if you later want a mobile app, it plugs into the same API with no backend changes.

Design decisions

DecisionRationale
UUID primary keysPrevents order ID enumeration attacks. Can generate IDs client-side before DB insert. Safer for public-facing order confirmation URLs.
jsonb for addressesAddress structure can vary (SA doesn't always have a province/state). jsonb avoids over-normalisation for something that's read far more than written.
price_at_purchase on order_itemsIf a product price changes after purchase, historical orders stay accurate. This is a legal requirement for invoicing.
product_snapshot jsonb on order_itemsIf a product is deleted or renamed, the order still has a record of what was actually purchased.
is_approved on reviewsReview moderation via CMS before any review goes live. Prevents spam or competitor abuse.
instagram_post_id on productsTracks which IG post a product was synced from. Prevents duplicate imports and allows re-sync.
fulfillment_type ENUM on ordersClient currently does both collection and delivery. ENUM enforces clean data and drives UI logic on both storefront and CMS order list.

Security Architecture

Guest checkout is not a security compromise — it's actually more secure in key ways. No passwords to leak, no user accounts to enumerate. The focus shifts to protecting the API, the payment flow, and the admin panel.

1
HTTPS Everywhere
Vercel and Render both provision SSL certificates automatically. All HTTP requests redirect to HTTPS. This is the baseline — non-negotiable and free.
2
CSRF Protection
Flask-WTF adds CSRF tokens to all form submissions including checkout and newsletter signup. State-changing endpoints reject requests without a valid token.
3
Rate Limiting
Flask-Limiter applies per-IP rate limits: 3/min on order creation, 10/min on cart writes, 60/min on product reads. Prevents scraping, spam orders, and brute-force on any endpoint.
4
Input Validation
Every field entering Flask is validated and sanitised. Email format, phone number, address length, quantity bounds. SQLAlchemy parameterised queries prevent SQL injection by default. No raw SQL.
5
Payment Security
Card data never touches our servers. PayFast and Yoco handle all card processing. Flask only receives a webhook notification after payment completes. HMAC signature validation on every webhook — reject anything that doesn't match.
6
Cart Session Model
Cart lives in localStorage (UUID session_id). At checkout, Flask validates the session_id exists and hasn't expired (7-day TTL). Stock is locked at order creation, not at add-to-cart — prevents overselling race conditions.
7
Security Headers
Flask-Talisman adds Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, Referrer-Policy automatically on every response.
8
Admin Auth (CMS only)
Better Auth secures the CMS. Admin accounts are email + password only. No self-registration — accounts are created manually. All admin API calls require a Bearer token. Tokens expire and rotate.
9
CORS Policy
Flask CORS configured to allow only the storefront and CMS origins. No wildcard. Any request from an unauthorised origin is rejected before it reaches any route handler.

Key Integrations

Instagram product sync flow

The client posts to Instagram as normal. In the CMS, they hit "Sync from Instagram". Flask calls the Meta Graph API, pulls the latest posts, and creates draft products for any post not already in the database. The client then reviews each draft, fills in price details, and publishes.

1
Client posts to IG
Normal post with product photo and caption
2
Admin hits Sync
CMS button triggers POST /api/admin/instagram/sync
3
Flask calls Graph API
Fetches media with caption + image URL
4
Check for new posts
Compare instagram_post_id — skip if already imported
5
Create draft products
is_active=false, images imported, caption → description
6
Admin reviews & publishes
Add price, category, variants. Set is_active=true to go live.

Instagram access token

Meta Graph API requires a long-lived token (60 days). Store in an environment variable, never in the database. Set up a cron job on Render to refresh the token every 45 days using the refresh endpoint. Token refresh failure should alert you via email immediately — a lapsed token breaks the sync silently.

Caption parsing strategy

Propose a caption convention to the client: product name on line 1, description on lines 2–3, hashtags at the end. Flask parses this structure to pre-fill product name and description. They can always edit in CMS before publishing. This saves significant manual work per product.

PayFast — South African standard

PayFast is the dominant SA gateway. Supports card, EFT, and SnapScan. The flow is: generate signed form → redirect user to PayFast hosted page → user pays → PayFast sends ITN (Instant Transaction Notification) webhook to Flask → Flask validates and updates order. Card data never leaves PayFast's servers.

Yoco — card popup option

Yoco keeps the user on your site via a popup overlay. Better UX than a full redirect. Load the Yoco JS SDK on the checkout page, charge the token in Flask, then handle the Yoco webhook. Yoco is strong for card-only transactions and has a clean SA merchant dashboard.

POPIA compliance notes (SA data law)

RequirementImplementation
Lawful basis for collecting dataData collected at checkout is necessary to fulfil the order — this is a lawful basis under POPIA. State clearly in the checkout form what the data is used for.
Privacy policyAdd a /privacy page to the storefront. Link it in the footer and at the checkout form. Keep it plain language.
Newsletter consentSeparate opt-in checkbox at checkout. Pre-ticked boxes are not allowed. Double opt-in email recommended.
Data retentionDefine a policy: order data retained for 5 years (accounting). Newsletter subscribers can unsubscribe at any time — add unsubscribe link to every email.
